## 0. Как читать этот документ

Стек разбит на три независимых слоя: **Frontend** (жёстко задан требованиями — React + Tailwind),
**Backend/API** (не зафиксирован в требованиях явно, выбирается сейчас) и **Инфраструктура**.
Для backend-слоя ниже — не единственно верный вариант, а обоснованная рекомендация с
альтернативами, потому что аналитика оставляет здесь свободу выбора (в отличие от фронтенда,
где React + Tailwind — это уже принятое решение, см. BR-9).

## 1. Важное уточнение скоупа: что вообще значит «API для него»

В брифе и в domain-description.md сказано: существующий бэкенд — **black-box источник истины**
по слотам, инструкторам, зонам и атомарности бронирования (R-004). Но при этом в
functional-requirements.md есть Must-требования, для которых в описанном «существующем бэкенде»
явно нет мест: профиль клиента (FR-33/34), оценки инструктора (FR-40/41 — хотя по одной из
версий data-model.md они вообще «вне скоупа», это внутреннее противоречие документов, см. §6),
Web Push подписки (FR-48), маппинг напоминаний (FR-45/46), Telegram-привязка (FR-2).

Это значит, что «API для клиента» — не проходной прокси к существующему бэкенду, а **отдельный
сервис (BFF — Backend For Frontend)**, который:

1. вызывает существующий бэкенд как источник истины по слотам/инструкторам/атомарности брони;
2. хранит и обслуживает то, что существующий бэкенд не покрывает (профиль в части UI-настроек,
   Web Push подписки, журнал уведомлений, при необходимости — оценки, если они не проектируются
   в существующей инфраструктуре);
3. закрывает контракт API (см. api-sequence.md) единым образом для фронтенда, чтобы фронт не
   знал про внутренние особенности старого бэкенда.

**Архитектурное решение, которое нужно принять до старта разработки** (сейчас это допущение):
собственный BFF-сервис с базой данных, а не «просто фронтенд + чужой API». Дальше стек описан
исходя из этого допущения — оно совпадает с тем, что уже нарисовано в api-sequence.md
(`POST /bookings`, `POST /bookings/{id}/cancel` — это эндпоинты **нашего** API, не чужого).

## 2. Frontend

### 2.1 Ядро — уже принятое решение

| Технология | Роль | Почему (требование) |
| :-- | :-- | :-- |
| **React 18+** | UI-библиотека | BR-9, NFR-4 — зафиксировано напрямую |
| **Tailwind CSS 3.x** | Стилизация | BR-9, NFR-4, 00-foundations.md §3 (структурные токены как Tailwind-утилиты) |
| **Vite** | Сборщик/дев-сервер | Быстрый HMR, из коробки PWA-плагин, легковеснее CRA/webpack, стандарт для SPA в 2025–2026 |
| **TypeScript** | Типизация | Не зафиксировано требованиями, но настоятельно рекомендуется: контракт API (data-model.md, api-sequence.md) уже строго типизирован — TS даёт этому контракту физическое воплощение и ловит рассинхрон фронта с API на этапе компиляции |

### 2.2 Роутинг и структура экранов

| Технология | Роль | Обоснование |
| :-- | :-- | :-- |
| **React Router v6/v7** | Роутинг между SCR-* экранами | Стандарт для SPA; поддерживает вложенные роуты (нужно для «модалка поверх экрана» — BS-001…BS-004 всегда открываются НАД конкретным SCR-*, см. 00-foundations §4.3) |
| Организация модалок через `<Outlet>` + parallel routes-подобный паттерн | Модалки BS-* | Чтобы Esc/бэкдроп/URL модалки работали предсказуемо и модалка не «съедала» состояние экрана-родителя (важно для правил §6.2 foundations — кто показывает toast после закрытия модалки) |

### 2.3 Данные с сервера и кэш

| Технология | Роль | Обоснование |
| :-- | :-- | :-- |
| **TanStack Query (React Query)** | Кэш серверного состояния, повторные запросы, инвалидация | Прямое попадание в NFR-24 (офлайн-кэш просмотра, пометка «данные могут быть неактуальны»), NFR-21 (перф.-цели p95), единый паттерн Loading/Content/Empty/Error из foundations §5 реализуется через `status` React Query «из коробки» |
| **Idempotency-Key генерация на клиенте** (`crypto.randomUUID()`) + мутации через `useMutation` | Создание/отмена брони | UC-1 требует обязательный `Idempotency-Key`; React Query позволяет держать один и тот же ключ на все ретраи одной попытки (NFR-9, UC-1 E5) |

> Zustand/Redux **не нужны** как отдельный стор глобального состояния: серверное состояние
> закрывает React Query, локальное UI-состояние (открыта ли модалка, шаг формы входа,
> выбор снаряжения на SCR-004) достаточно держать в `useState`/`useReducer`/URL-параметрах —
> это соответствует принципу «минимум сложности» при ограниченном бюджете (BR-7).

## 3. Backend / API

> Ниже — рекомендация «по умолчанию». Она не единственно возможная, но выбрана с учётом:
> жёсткого срока ~2 месяца (BR-7/NFR-15), ограниченного бюджета (NFR-16), необходимости шарить
> типы/схемы с TypeScript-фронтендом, и требований к атомарности (NFR-8/9) и идемпотентности (UC-1).

### 3.1 Язык и рантайм

| Технология | Обоснование |
| :-- | :-- |
| **Node.js (LTS) + TypeScript** | Общий язык с фронтендом → можно шарить типы контракта API (интерфейсы `Booking`, `Slot`, коды ошибок из матрицы ошибок UC-1) через общий пакет/monorepo; большая экосистема готовых интеграций (Telegram Bot API, Web Push, SMS-провайдеры); при команде из 1–3 разработчиков на 2 месяца — меньше когнitивного переключения между языками |

**Альтернатива:** Python (FastAPI) — тоже быстрый старт и хорошая типизация через Pydantic,
разумный выбор, если в команде сильнее Python-компетенция. Дальше по документу — вариант на
Node.js, но все архитектурные решения (BFF, Redis для идемпотентности, Postgres) переносятся
на FastAPI без изменений.

### 3.2 Web-фреймворк

| Технология | Обоснование |
| :-- | :-- |
| **Fastify** | Быстрее и легче NestJS по порогу входа при 2-месячном сроке; встроенная валидация схем (JSON Schema/TypeBox), из коробки хорошо ложится на OpenAPI-генерацию |

**Альтернатива:** **NestJS** — если ожидается, что после MVP команда вырастет и появится
роль/админ-API (Phase 2) — модульная архитектура Nest окупится на масштабировании. Для
one-shot MVP с фиксированным скоупом (только роль «Клиент», FR-28) Fastify даёт то же самое
с меньшим количеством boilerplate.

### 3.3 База данных

| Технология | Роль | Обоснование |
| :-- | :-- | :-- |
| **PostgreSQL** | Основное хранилище (Client-профиль, Booking, Rating, NotificationLog, WebPushSubscription) | Нужны транзакции и блокировки (`SELECT … FOR UPDATE` или `SERIALIZABLE`) для атомарной проверки свободных мест — это прямое требование NFR-8/NFR-22 (нагрузочный тест «гонка бронирований»), даже если финальная проверка мест физически идёт через существующий бэкенд — сама BFF-транзакция (запись Booking + ответ) должна быть atomic |
| **Prisma ORM** | Доступ к БД, миграции, типы | Генерирует TS-типы из схемы — снова та же логика «один источник истины для типов», плюс простые миграции для короткого проекта |

### 3.4 Идемпотентность и rate-limit

| Технология | Роль | Обоснование |
| :-- | :-- | :-- |
| **Redis** | (а) хранение `Idempotency-Key` → закэшированный ответ на ~24ч; (б) rate-limit для OTP (NFR-19: не чаще раза в 30–60 с, ≤5 запросов в час); (в) blacklist отозванных refresh-токенов (NFR-18) | Все три задачи — классические TTL-операции, Postgres для них избыточен и медленнее под нагрузкой пиковых часов (NFR-7, NFR-22) |

### 3.5 Аутентификация

| Технология | Роль | Обоснование |
| :-- | :-- | :-- |
| **SMS OTP через провайдера** (напр. SMS Aero, SMSC.ru, или Twilio Verify для международного охвата) | Отправка и валидация одноразового кода | FR-1, NFR-19; берём готовый провайдер, а не пишем свою SMS-шлюз-интеграцию — вне зоны компетенции и бюджета проекта |
| **Telegram Bot API** (библиотека `grammy` или `telegraf`) | Вход через Telegram + дублирование уведомлений | FR-2, NFR-26 |
| **JWT (`jose`)** — access (~15 мин, в памяти клиента) + refresh (~30 дней, httpOnly Secure SameSite=Strict cookie) | Сессии | NFR-18 — сформулировано в требованиях буквально до деталей cookie-флагов, стек просто реализует это как есть |
| **CSRF-защита на мутирующих запросах** (double-submit cookie или заголовок) | Т.к. используется cookie-based refresh | NFR-18 |

### 3.6 Уведомления и фоновые задачи

| Технология | Роль | Обоснование |
| :-- | :-- | :-- |
| **web-push (npm)** | Отправка Web Push по VAPID-ключам | NFR-17 |
| **Telegram Bot API** (см. выше) | Дублирующий канал, основной по надёжности | NFR-17, NFR-26 |
| **BullMQ (на Redis)** | Очередь + шедулер для напоминаний «за 24ч / за 3ч» и рассылки при отмене скалодромом | FR-45/46 требуют триггеров по времени и «разом всем записанным» — очередь с задержанной доставкой (`delay`) и retry на сбой канала (NFR-26: ошибка Telegram не блокирует остальное) — это ровно то, что даёт BullMQ «из коробки» |

### 3.7 Документация и контракт API

| Технология | Роль | Обоснование |
| :-- | :-- | :-- |
| **OpenAPI 3.x (redocly.yaml, уже упомянут в api-sequence.md)** | Контракт API как источник истины | Уже зафиксировано в проектной структуре (`api/redocly.yaml`, `api/bookings/api.yaml`) — стек лишь генерирует из него типы (`openapi-typescript`) для фронта и валидирует запросы на бэке (`fastify-openapi-glue` или ручная TypeBox-схема, синхронизированная со спекой) |

## 4. Интеграция с существующей инфраструктурой

| Вопрос | Рекомендация |
| :-- | :-- |
| Протокол вызова существующего бэкенда | REST/HTTP, если существующий бэкенд его поддерживает (наиболее вероятный вариант при отсутствии иных указаний); если существующий бэкенд асинхронный/событийный — потребуется отдельное уточнение у владельца инфраструктуры до начала разработки BFF |
| Обработка недоступности существующего бэкенда | Circuit breaker / таймаут ~10 с на исходящий вызов (симметрично клиентскому таймауту NFR-24), единообразный проброс `5xx` клиенту по паттерну Error/Retry |
| Кэширование справочников (инструкторы, зоны) | Короткий TTL-кэш на стороне BFF (Redis, 30–60 с) — снижает нагрузку на существующий бэкенд в часы пик (NFR-7), не противоречит NFR-9 (согласованность), так как это read-only данные |

## 5. Инфраструктура и деплой

| Слой | Технология | Обоснование |
| :-- | :-- | :-- |
| Хостинг фронтенда | **Статический хостинг/CDN** (Cloudflare Pages, Vercel или обычный Nginx + сборка Vite) | SPA на статике — дешевле и быстрее по TTFB, чем через Node-сервер; критично для NFR-21 (p95 < 2.5 c) |
| Хостинг BFF | **Docker-контейнер** на VPS или PaaS (Render/Railway/облачный провайдер РФ, если есть требования по локализации ПДн) | Учебный/тестовый проект — не обязательно оптимизировать под масштаб, важнее скорость деплоя |
| CI/CD | **GitHub Actions** (lint → typecheck → test → build → deploy) | Стандарт, бесплатный тир достаточен для проекта такого размера |
| TLS | Let's Encrypt / встроенный TLS PaaS | NFR-18, NFR-20 требуют TLS 1.2+ везде |
| Мониторинг ошибок | **Sentry** (frontend + backend) | Нет явного NFR про мониторинг, но NFR-9/NFR-22 (целостность при сбоях) практически нереализуемы без наблюдаемости в проде |
| Логи | Структурированные JSON-логи (`pino` — совместим с Fastify) с маскированием телефона/OTP | NFR-20 — прямой запрет логировать полный номер и код |

## 6. Открытые вопросы, которые стек не решает сам

Технологический выбор не может закрыть противоречия в самих требованиях — их нужно разрешить
до/во время реализации:

1. **Хранение оценок инструктора.** functional-requirements.md держит FR-40/41 как Must для
   клиентского MVP (после переноса из user-stories.md — customer-questions.md №20 говорит, что
   оценки **вынесены в Phase 2**, а functional-requirements.md всё ещё содержит их как Must).
   Нужно свериться с заказчиком, какая версия актуальна — от этого зависит, нужна ли таблица
   `Rating` в БД BFF на старте или её можно не создавать.
2. **Формат вызова существующего бэкенда** (REST/GraphQL/gRPC/очередь) нигде не описан — стек
   BFF предполагает REST по умолчанию, но это допущение, не факт.
3. **data-model.md / api-sequence.md / er-model-and-sequences.md** содержат сущности из другого
   домена (поход/маршрут с геометрией, доски для скалолазания, до 3 мест в одной брони) —
   похоже на не до конца адаптированный шаблон другого проекта (см. также пометку-шаблон в
   design-review.md). Прежде чем реализовывать модель БД по этим файлам буквально, стоит сверить
   их с domain-description.md/functional-requirements.md, где бронь — **на одного клиента**, без
   гостей, без карты и геометрии маршрута.
