# Описание домена — клиентское приложение скалодрома «Вертикаль»

## 1. Контекст и границы скоупа

Предметная область — сервис записи клиентов на групповые тренировки в скалодроме.
Разрабатывается **только клиентская часть**: веб-приложение (React + Tailwind CSS) и API для него.

**Платформа.** Клиентская часть — адаптивное веб-приложение (SPA), открывается в браузере с
телефона, планшета или компьютера без установки из магазина приложений. Технологический
стек — React + Tailwind CSS. Интерфейс проектируется mobile-first (основной сценарий —
браузер на телефоне в зале скалодрома), но должен корректно работать и на десктопе.

Уже реализовано в существующей инфраструктуре (вне скоупа):
- бэкенд, хранение данных, формирование расписания;
- интерфейсы инструктора и владельца/администратора (админка);
- атомарная проверка свободных мест при бронировании — гарантия «без двойных броней» обеспечивается бэкендом, клиент лишь корректно обрабатывает отказ.

Клиентское приложение **потребляет** слоты, зоны/форматы и инструкторов через API, но не создаёт и не редактирует их.

Каноническая схема данных — контракт API; бэкенд отдаёт все нужные поля (тариф на прокат, зону/формат тренировки, адрес скалодрома, свободный прокатный фонд и т.д.). Исторических данных и легаси-схемы нет (проект учебный/тестовый).

Скалодром — одна площадка (один адрес), мультилокационность не требуется.

## 2. Роли

В текущей поставке реализуется только роль **Клиент**. Интерфейсы инструктора и владельца существуют отдельно и в скоуп не входят.

### Аутентификация
Вход клиента — по номеру телефона через SMS-код, а также через Telegram.

## 3. Ключевые сущности

### 3.1. Клиент
- Записывается на тренировки, отменяет брони.
- Может иметь **несколько активных броней одновременно**, ограничений по количеству нет.
- Имеет флаг «новичок» (приходит из бэкенда/профиля), определяющий доступ к новичковым тренировкам.
- Оставляет оценку инструктору после посещения тренировки.
- Статус «постоянный клиент» и связанные с ним бонусы — видны и управляются **только в админке**, в клиентском приложении не отображаются.

### 3.2. Тренировка (слот)
Объект, приходящий из бэкенда через API целиком; приложение его не создаёт и не редактирует.

Атрибуты:
- дата и время начала;
- длительность (~1.5 часа);
- зона/формат: болдеринг с инструктажем для новичков / трассы с верёвкой для опытных;
- назначенный инструктор (включая средний рейтинг — см. 3.3);
- вместимость: до 16 человек в обычной группе, до 8 — в новичковой;
- количество свободных мест;
- флаг «для новичков» (ограничивает доступ, а не только количество мест).

**Горизонт показа:** по умолчанию клиенту показываются слоты на ближайшие **7 дней**, весь диапазон открыт сразу (не поэтапно). Более длинный период — через фильтр дат. Если слотов нет — понятный empty state («Пока нет доступных тренировок»).

### 3.3. Инструктор
В клиентской части — источник данных для отображения, без отдельного интерфейса.

Атрибуты, видимые клиенту:
- имя;
- **средний рейтинг** (агрегированное значение по оценкам клиентов, 1–5) — отображается в карточке/списке тренировок при выборе слота, до записи. Расчёт агрегата — на стороне бэкенда, приложение только отображает готовое значение.
- Если у инструктора ещё нет оценок — отображается явный empty-state («Нет оценок»), а не 0 или пустое поле.

### 3.4. Бронь (запись)
Связь клиента и слота.

Статусы:
- активна;
- отменена клиентом (вовремя);
- отменена клиентом поздно;
- отменена скалодромом;
- завершена.

**Правила отмены:**
- Порог поздней отмены — **2 часа** до начала тренировки.
  - Отмена ≥ 2 часов до начала — «чистая» отмена, место освобождается.
  - Отмена < 2 часов — фиксируется в истории брони/клиента (флаг «поздняя отмена»); никаких блокировок или лимитов на стороне клиентского приложения не применяется, данные лишь передаются через API для дальнейшего использования (например, в админке).
- Листа ожидания **нет** — при отсутствии мест клиент просто не может записаться.

**Отмена тренировки скалодромом** (в т.ч. по профилактике/проверке снаряжения):
- бронь не удаляется, получает статус «Отменена скалодромом» с указанием причины;
- клиенту приходит уведомление (браузерный Web Push и/или Telegram);
- повторная запись на отменённый слот запрещена;
- приложение **не предлагает** альтернативный слот — клиент ищет новое время самостоятельно;
- само действие отмены слота выполняется в существующей инфраструктуре, вне скоупа клиентского приложения.

**Снаряжение:** при создании брони клиент выбирает — своё снаряжение или прокат. Выбор делается **на уровне каждой конкретной брони**, не сохраняется как настройка профиля по умолчанию.

### 3.5. Прокатное снаряжение
- Ограниченный фонд, есть тариф на аренду.
- Доступность прокатного инвентаря проверяется **отдельно** от свободных мест в группе.
- Если мест в тренировке достаточно, но прокатного инвентаря на всех не хватает — запись **не блокируется**, но клиенту показывается постоянная информационная плашка об ограниченности инвентаря на всех этапах оформления брони (выбор слота → подтверждение → экран брони).
- Оплата проката — информационная (тариф отображается, оплата на месте наличными/переводом), клиент дополнительно уведомляется об этом условии.

### 3.6. Оценка
- Разовая оценка инструктора после посещения тренировки.
- Шкала 1–5 звёзд + текстовый комментарий.
- Оценку нельзя изменить после отправки.
- Участвует в расчёте среднего рейтинга инструктора (см. 3.3).

### 3.7. Уведомление
Каналы: **браузерный Web Push (если пользователь дал разрешение) и дублирование в Telegram.**
Telegram — основной, гарантированно доступный канал (не все браузеры/устройства поддерживают
Web Push, особенно на iOS); Web Push — дополнительный канал «до открытия вкладки».

Триггеры:
- напоминание о предстоящей тренировке — за **1 день** и за **3 часа** до начала;
- уведомление об отмене тренировки скалодромом (с причиной);
- уведомление об условиях оплаты проката при бронировании с прокатом.

## 5. Открытые/зависимые от бэкенда моменты
- Расчёт среднего рейтинга инструктора — ответственность бэкенда, клиент получает готовое значение через API.
- Атомарность бронирования (защита от двойной записи) — реализуется на бэкенде, вне скоупа.
- Формирование и редактирование расписания, зон/форматов, назначение инструкторов — вне скоупа, приложение только читает.
