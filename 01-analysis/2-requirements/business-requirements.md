# Бизнес-требования

> Этап 2. Зачем бизнесу продукт: цели, метрики успеха, ограничения верхнего уровня.

> **Скоуп — клиентское веб-приложение (React + Tailwind CSS) и API для него.** Бизнес-цели, обслуживаемые
> существующей инфраструктурой (картина для владелицы, интерфейс инструктора, формирование
> расписания), в этот список не входят.

> **О нумерации требований.** ID требований (BR-NN, M-NN) сохраняются стабильными для
> трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть
> ранее рассматривавшихся требований — лист ожидания, программа лояльности для постоянных
> клиентов, онлайн-оплата — вынесена за пределы текущей поставки или в развитие). Нумерация
> намеренно не нормализуется, чтобы не ломать связи в таблице-источнике и смежных документах.

> **Роли в текущей поставке.** В текущую поставку входит только роль **Клиент**
> (самостоятельная запись на тренировки через веб-приложение). Интерфейсы инструктора и
> администратора/владелицы — **вне скоупа текущей поставки**: они обеспечиваются существующей
> инфраструктурой. Это осознанное решение в рамках MVP-подхода и жёсткого срока: продукт
> фокусируется на клиентском сценарии, который снимает основную боль (хаос записи и двойные
> брони), а серверные/ролевые функции остаются на стороне существующего бэкенда.

| BR-1 | Устранить ручную путаницу и двойные брони, возникающие при ведении записи через Telegram и бумажную тетрадь. Учёт записей должен вестись в единой системе, исключающей конфликты по местам в группе. | Must | Бриф (письмо Оли, п. о хаосе в часы пик и двойных записях); Описание домена → «Контекст и границы скоупа» |

## Метрики успеха

> Целевые значения — разумные ориентиры для MVP; точные пороги фиксируются как допущение и при
> появлении реальных данных уточняются.

| ID | Метрика | Целевой ориентир | Связь с требованиями |
| :-- | :-- | :-- | :-- |
| M-1 | Доля записей, сделанных клиентами онлайн самостоятельно (от общего числа записей за сезон). | ≥ 70% к концу первого сезона | BR-2 |
| M-2 | Число двойных броней / конфликтов по местам. | 0 (полное устранение) | BR-1 |
| M-4 | Доля поздних отмен (< 2 часов до начала) от общего числа записей. | Снижение относительно текущего уровня (целевой ориентир ≤ 10%) | BR-5 |
| M-5 | Доля тренировок с проставленной клиентом оценкой инструктора. | ≥ 50% посещённых тренировок к концу первого сезона | BR-10 |
| M-6 | Готовность к старту сезона. | Запуск в срок ~2 месяцев с критичным набором функций (просмотр слотов, запись, отмены, уведомления об отмене скалодромом) | BR-7 |

## Ограничения верхнего уровня

- **Срок и бюджет.** Жёсткая цель — начало нового сезона (~2 месяца); бюджет ограничен →
  строго MVP-подход, «навороты» откладываются. (Бриф; Описание домена → «Контекст и границы
  скоупа»)
- **Оплата офлайн на старте.** Онлайн-оплата (в т.ч. проката) вне границ скоупа. (Ответы
  заказчика № 10)
- **Веб, mobile-first.** Клиентское приложение — веб-приложение (React + Tailwind CSS),
  адаптивное, в первую очередь под мобильный браузер (используется с телефона в зале), без
  установки из магазина приложений; данные получает из существующего бэкенда через API,
  слоты/расписание/зоны/инструкторы не создаются и не редактируются на стороне клиента.
  (Бриф)
- **Лист ожидания не реализуется.** Осознанно исключён из скоупа — при отсутствии мест клиент
  просто не может записаться. (Ответы заказчика № 3)
- **Программа лояльности / отметка постоянных клиентов — вне скоупа клиента.** Управляется
  только в существующей админке, в клиентском приложении не отображается. (Ответы заказчика
  № 14)
- **Вне скоупа.** Функции инструктора/владелицы, CRUD расписания и админка обеспечиваются
  существующей инфраструктурой. Онлайн-оплата, лист ожидания, программа лояльности —
  направления развития (Phase 2 / backlog). (Описание домена → «Контекст и границы скоупа»)
