# Нефункциональные требования

> Этап 2. Как система должна работать: производительность, надёжность, безопасность,
> доступность, удобство. Частая ошибка ИИ — путать ФТ и НФТ.

> **Скоуп — клиентское веб-приложение (React + Tailwind CSS) и API для него.** НФТ относятся к
> клиентскому приложению; функции инструктора/владелицы/админки обеспечиваются существующей
> инфраструктурой.

> **О нумерации ID.** Разрывы в нумерации NFR-NN — следствие сужения скоупа до клиентского
> приложения (часть ранее рассматривавшихся требований, например к онлайн-оплате и картографии,
> не применима к проекту или вынесена в развитие). Существующие ID сохранены, чтобы не ломать
> внешние ссылки; новые требования нумеруются продолжением схемы. Сквозная нормализация
> нумерации не выполняется намеренно.

> **Целевая платформа.** Продукт — адаптивное веб-приложение (SPA) на **React + Tailwind CSS**,
> открывается в браузере без установки, распространяется как обычный сайт (см. NFR-4).
> Требования к уведомлениям, хранению токена сессии и a11y сформулированы исходя из веб-стека,
> а не нативного мобильного приложения.

| NFR-17 | Уведомления | Каналы уведомлений в MVP — браузерный **Web Push** (Notification API + Service Worker, регистрация подписки через `PushManager`) **и** дублирующее сообщение в Telegram (см. Ответы заказчика № 11). Telegram — основной, гарантированно доступный канал: Web Push на iOS Safari требует установки сайта на домашний экран (Add to Home Screen) и поддерживается только начиная с iOS 16.4+, поэтому на части устройств Web Push может быть недоступен в принципе. Подписка Web Push (`endpoint`, ключи) регистрируется на бэкенде после явного разрешения пользователя в браузере; при отзыве разрешения, недоступности API на устройстве или невалидной подписке push тихо не отправляется (без ошибки для пользователя), Telegram-дублирование (если аккаунт привязан) продолжает работать независимо. **Тайминг напоминаний в MVP — за 24 ч и за 3 ч до начала тренировки** (`reminder_hours = [24, 3]`), согласовано с заказчиком (Ответ № 12); конкретные часы клиент берёт из ответа API, не хардкодит. | Средний | Ответы заказчика № 11–12; решение о смене архитектуры на веб |
| NFR-18 | Безопасность / Сессия | Сессия — JWT с парой токенов: короткоживущий **access** (срок жизни ~15 мин) и долгоживущий **refresh** (срок жизни ~30 дней). Для веб-архитектуры **refresh-токен хранится в `httpOnly` + `Secure` + `SameSite=Strict` cookie** (недоступен из JavaScript, что закрывает основной вектор XSS-кражи токена); **access-токен хранится только в памяти приложения** (JS-переменная/React state), не в `localStorage`/`sessionStorage` и не в обычной cookie, и теряется при полной перезагрузке вкладки — в этом случае клиент прозрачно для пользователя обновляет его через refresh-cookie при первом запросе к API. Access автоматически обновляется по refresh без участия пользователя; повторный вход (SMS OTP или Telegram) требуется только при истечении/отзыве refresh. **401-flow:** при `401` от API клиент дёргает эндпоинт обновления (refresh-cookie уходит автоматически браузером); при неуспехе — сбрасывает access из памяти и переводит пользователя на экран входа. Весь обмен с API — только по TLS (HTTPS, TLS 1.2+), с CSRF-защитой на мутирующих запросах (например, проверка кастомного заголовка/токена, так как используется cookie-based refresh). | Высокий | описание домена (аутентификация); решение о смене архитектуры на веб |
| NFR-19 | Безопасность / Антифрод OTP | Отправка SMS-кода ограничена rate-limit'ами: не более ~1 запроса кода в 30–60 с на номер и не более ~5 запросов в час на номер/устройство; ограничено число попыток ввода кода (напр. ≤5) до временной блокировки; код имеет конечный TTL. При исчерпании лимитов пользователю показывается понятное сообщение с указанием времени до повторной попытки. | Высокий | Ответы заказчика № 13 |

| NFR-21 | Производительность (измеримая) | Целевые метрики при «нормальной сети» (4G/Wi-Fi, ≥5 Мбит/с): получение списка тренировок и открытие карточки слота — **p95 < 2.5 с**; создание и отмена брони (`createBooking`/`cancelBooking`) — **p95 < 1.5 с**. | NFR-22 | Производительность / Нагрузка | Расчётный пик — часы пик записи (по брифу — вечера и выходные), оценочно до нескольких десятков одновременных активных пользователей (предположение). | NFR-23 | Доступность (Availability) | Целевая доступность — **99%** в рабочие часы скалодрома в течение сезона. Окно измерения и все временные правила (в т.ч. порог поздней отмены — 2 часа) трактуются в **локальной таймзоне скалодрома**. Конкретные рабочие часы и границы сезона фиксируются как **обязательный pre-launch config** с владелицей; **критерий приёмки:** значения заданы и проверены до запуска. Из расчёта доступности **исключаются**: плановые работы (с предварительным уведомлением) и недоступность внешней инфраструктуры (SMS/OTP-провайдер, Telegram Bot API, браузерные push-сервисы). **Деградация:** при отказе Web Push (или его отсутствии на устройстве, см. NFR-17) — бронирование и отмена работают без браузерных напоминаний, дублирование в Telegram продолжает работать (если доступно), и наоборот. Уточняет NFR-10. | Средний | NFR-10 |
| NFR-24 | Надёжность / Offline | Просмотр уже загруженных данных (списка слотов, своих броней) доступен офлайн из кэша **Service Worker** (Cache API) с явной пометкой «данные могли устареть». Мутации (создание/отмена брони, отправка оценки) офлайн **запрещены** — показывается понятное сообщение о необходимости подключения. Единый паттерн Error/Retry для всех сетевых ошибок. Таймаут сетевого запроса — ~10 с. При неопределённом результате создания брони (таймаут/обрыв) клиент повторяет запрос с тем же `Idempotency-Key`, что исключает дубли. | Средний | описание домена |
| NFR-25 | Доступность (A11y) | Приёмочные критерии доступности (проверяемые при приёмке): контраст текста и ключевых элементов соответствует WCAG 2.1 уровня AA; минимальный размер интерактивных touch-target ≥ 44px (Tailwind-утилиты для отступов/размеров кнопок); поддержка увеличения масштаба страницы браузером (zoom до 200%) без потери функциональности и горизонтального скролла; вся ключевая функциональность (просмотр слотов, запись, отмена, оценка) доступна **с клавиатуры** (Tab/Shift+Tab, Enter/Space, видимый focus-ring); семантическая разметка и ARIA-атрибуты для экранных читалок (VoiceOver, NVDA, TalkBack — в зависимости от браузера/ОС пользователя); статус брони и доступность мест не передаются одним только цветом; интерфейс остаётся работоспособным и читаемым на малых экранах (см. NFR-4). Уточняет NFR-1. | Средний | NFR-1 |
| NFR-26 | Интеграции / Telegram | Дублирование уведомлений в Telegram выполняется через Telegram Bot API после того, как клиент привязал Telegram-аккаунт (см. FR-2). > **Вне скоупа НФТ.** Требования к картографии/геометрии маршрутов не применимы — скалодром
> имеет одну площадку с известным адресом (см. Ответы заказчика № 15), интерактивная карта не
> требуется. Требования к онлайн-платежам (PCI DSS и т.п.) не формулируются — оплата на старте
> офлайн (см. NFR-16).
