# Use cases / сценарии

> Этап 2. Сценарии использования. Обязательно прорабатываем **альтернативные потоки** —
> ИИ любит их забывать.
>
> **Скоуп: клиентское приложение и API для него.** Слоты, зоны/форматы и инструкторы поступают
> из существующего бэкенда через API (read-only).

## UC-1: Запись на тренировку

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован; в расписании есть хотя бы один слот со свободными местами.
- **Триггер:** Клиент хочет забронировать место на тренировке.

> **Ограничение доступа к новичковым тренировкам.** Запись на слот с признаком «для новичков»
> доступна только клиентам с флагом «новичок» в профиле (приходит из API). Ограничение сверх
> вместимости группы (см. FR-16).
>
> **Бизнес-решение по идемпотентности.** Клиент **обязан** передавать `Idempotency-Key` при
> создании брони. Повтор той же операции (тот же ключ и то же тело запроса) даёт идентичный ответ
> и не создаёт дубль брони. Повтор с тем же ключом, но другим телом — ошибка.
>
> **Время и источник истины.** Время хранится и передаётся в UTC, отображается в локальной зоне.
> Сервер — источник истины по времени и статусам; при расхождении клиентского прогноза и
> серверного ответа применяется серверный статус.

### Основной поток
1. Клиент открывает список тренировок.
2. (Опц.) Применяет фильтры: дата/период, зона/формат, инструктор.
3. Открывает карточку слота и просматривает параметры (дата/время, зона/формат, признак «для новичков», инструктор и его рейтинг, цена, свободно мест, доступность прокатного снаряжения).
4. Выбирает вариант снаряжения: своё / прокат.
5. Подтверждает запись. Клиент формирует и передаёт `Idempotency-Key` для этой операции.
6. Система проверяет: (а) флаг «новичок» — если слот новичковый; (б) лимит мест слота (до 16 в группе, до 8 в новичковой); (в) **отдельно** доступность прокатного снаряжения, если выбран прокат; создаёт бронь, уменьшает счётчики свободных мест/прокатного фонда.
7. Система показывает подтверждение с тарифом (оплата офлайн) и добавляет бронь в «Мои бронирования».

### Альтернативные потоки
- **A1. Своё снаряжение.** На шаге 4 прокатный фонд не затрагивается; уменьшается только число свободных мест слота.
- **A2. Прокатное снаряжение, но фонд ограничен.** На шаге 3 клиент видит предупреждающую плашку о возможной нехватке инвентаря; запись при этом не блокируется (см. E2).
- **A3. Фильтр не дал результатов.** На шаге 2 список пуст — система предлагает сбросить/изменить фильтры.

### Исключения / ошибки
- **E1. Не хватает мест в группе.** Запрошено больше, чем свободно мест → запись отклоняется, показывается актуальное число свободных мест.
- **E2. Не хватает прокатного снаряжения.** Прокат запрошен, а свободного инвентаря нет → система предлагает выбрать «своё снаряжение»; запись без проката остаётся доступной.
- **E3. Слот для новичков, а флаг не установлен.** Попытка записи отклоняется с понятным сообщением о том, что тренировка предназначена для новичков.
- **E4. Гонка запросов / слот заполнился параллельно.** На шаге 6 места уже заняты другим клиентом → запись отклоняется без овербукинга и двойной брони (NFR-8), карточка слота обновляется.
- **E5. Сетевой сбой / неопределённый результат при подтверждении.** Бронь не создаётся частично; при неопределённом результате клиент повторяет операцию с **тем же** `Idempotency-Key`, повтор не приводит к задвоению (NFR-9). Мутации (создание брони) офлайн запрещены — при отсутствии сети показывается понятное сообщение, кнопка подтверждения недоступна.
- **E6. Слот отменён скалодромом или уже стартовал.** Слот к моменту подтверждения отменён скалодромом или тренировка началась → запись отклоняется с соответствующим кодом (`slot_cancelled` / `slot_started`).

### Матрица ошибок ключевых операций

Канон таксономии — коды API. Источник локализованного текста — клиентское приложение (по коду ошибки и полям `details`).

| Код | HTTP | Обязательные `details` | UX-реакция | Retry |
| :-- | :-- | :-- | :-- | :-- |
| `slot_full` | 409 | `available_seats` | Показать актуальное число свободных мест, обновить карточку | Нет |
| `rental_unavailable` | 409 | `available_rental_items` | Предложить выбрать «своё снаряжение» вместо проката | Нет (изменить выбор снаряжения) |
| `beginner_flag_required` | 403 | `slot_id` | Сообщить, что тренировка только для новичков (флаг не установлен) | Нет |
| `double_booking` | 409 | `booking_id` | Сообщить, что бронь на слот уже есть; перейти в «Мои бронирования» | Нет |
| `slot_cancelled` | 410 | `slot_id` | Сообщить об отмене слота скалодромом, убрать CTA записи | Нет |
| `slot_started` | 422 | `slot_id`, `start_at` | Сообщить, что тренировка началась/прошла, запись недоступна | Нет |
| `invalid_code` | 400/401 | — | Сообщение «неверный код», поле OTP остаётся активным | Да (ввод заново) |
| 401 (unauthorized) | 401 | — | Очистка токена, переход на авторизацию | Да (после входа) |
| 403 (forbidden) | 403 | — | Сообщение об отсутствии прав; действие недоступно | Нет |
| 429 (rate limit) | 429 | `retry_after` | Сообщение «слишком много попыток», отложенный повтор | Да (после `retry_after`) |
| 5xx (server error) | 5xx | — | Единый Error/Retry; для брони — повтор по тому же `Idempotency-Key` | Да |

### Постусловия
- При успехе: создана бронь клиента; счётчики слота и прокатного фонда уменьшены; бронь видна в «Мои бронирования».
- При ошибке: состояние слота не изменилось.

## UC-2: Отмена записи

- **Актёры:** Клиент
- **Предусловия:** У клиента есть предстоящая бронь; тренировка ещё не стартовала.
- **Триггер:** Клиент решает отменить участие.

> **Бизнес-решение по объёму отмены.** Отмена выполняется **только целиком**: отменяется вся бронь со всеми местами и прокатным снаряжением. Частичная отмена не поддерживается.
>
> **Время и источник истины.** Время хранится и передаётся в UTC, отображается в локальной зоне. Сервер — источник истины по времени и по статусу отмены. Граница политики: **ровно 2 часа до старта = ещё ранняя (бесплатная) отмена**. При расхождении клиентского прогноза (ранняя/поздняя) и серверного ответа применяется **серверный** статус.

### Основной поток
1. Клиент открывает «Мои бронирования» и выбирает предстоящую запись.
2. Нажимает «Отменить» (отмена всей брони целиком).
3. Сервер определяет время до старта (источник истины — серверное время).
4. **Ранняя отмена (≥2 ч до старта):** бронь помечается «отменена»; забронированное место и прокатное снаряжение возвращаются в слот.
5. Система показывает подтверждение отмены.

### Альтернативные потоки
- **A1. Поздняя отмена (<2 ч до старта).** На шаге 4 бронь помечается «поздняя отмена»; место НЕ освобождается; денежных штрафов нет; клиенту показывается пояснение политики 2 часов.
- **A2. Расхождение клиентского прогноза и сервера.** Клиент показал «ранняя отмена», а сервер вернул «поздняя» (или наоборот) → применяется и отображается серверный статус.

### Исключения / ошибки
- **E1. Тренировка уже стартовала/прошла (`slot_started`).** Кнопка отмены недоступна; бронь нельзя отменить.
- **E2. Бронь уже отменена (`already_cancelled`).** Повторная отмена не выполняется; показывается текущий статус.
- **E3. Отмена офлайн / неопределённый результат.** Мутация отмены офлайн запрещена — при отсутствии сети показывается понятное сообщение, кнопка недоступна; при неопределённом результате применяется единый Error/Retry, финальный статус берётся с сервера. Коды ошибок и UX-реакции — по матрице ошибок UC-1.

### Постусловия
- Статус брони обновлён («отменена» / «поздняя отмена»).
- При ранней отмене место и прокатное снаряжение брони снова доступны другим клиентам.

## UC-3: Просмотр и фильтрация списка тренировок

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет найти подходящую тренировку.

### Основной поток
1. Клиент открывает список тренировок. По умолчанию показываются слоты на **ближайшие 7 дней** (дефолт API: `date_from = now`, `date_to = now + 7 дней`; больший период — явным фильтром дат), отсортированные по времени старта (по возрастанию).
2. Заполненные (`free_seats = 0`) и отменённые скалодромом слоты отображаются в списке с пометкой «Мест нет» / «Отменена скалодромом» и неактивной (disabled) CTA записи.
3. (Опц.) Задаёт фильтры: дата/период, зона/формат, инструктор.
4. Система показывает отфильтрованный список с ключевыми параметрами каждого слота, сохраняя сортировку по времени старта.

### Альтернативные потоки
- **A1. Сброс фильтров.** Клиент очищает фильтры → возвращается дефолтная выдача (слоты на ближайшие 7 дней, без фильтров).
- **A2. Семантика комбинирования фильтров.** Внутри одной группы значения комбинируются по «ИЛИ» (`route_type`/`instructor_id` — массивы), между разными группами — по «И». Границы периода дат включительные; ввод и трактовка дат — в локальной зоне клиента. Активные фильтры отображаются клиенту.
- **A3. Пустой справочник инструкторов.** Если список инструкторов пуст, фильтр по инструктору скрывается.

### Исключения / ошибки
- **E1. Нет слотов под условия фильтра.** Показывается пустое состояние с подсказкой изменить/сбросить фильтры.
- **E2. Нет расписания вовсе.** В горизонте выдачи нет ни одного предстоящего слота → показывается пустое состояние «Пока нет доступных тренировок».
- **E3. Просмотр офлайн.** При отсутствии сети показывается ранее загруженный кэш списка с пометкой «данные могли устареть»; единый Error/Retry для повторной загрузки.

### Постусловия
- Клиент видит актуальный (или помеченный как устаревший) список тренировок; данные расписания не изменяются.

## UC-4: Оценка инструктора после тренировки

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован; у клиента есть завершённая (посещённая) тренировка, ещё не оценённая им ранее.
- **Триггер:** Клиент хочет оставить отзыв об инструкторе после тренировки.

### Основной поток
1. Клиент открывает завершённую бронь в «Мои бронирования».
2. Система показывает форму оценки: шкала 1–5 звёзд и поле комментария.
3. Клиент выбирает количество звёзд и (опц.) вводит комментарий.
4. Клиент отправляет оценку.
5. Система сохраняет оценку, привязывая её к брони и инструктору; форма становится недоступной для повторного изменения.
6. Средний рейтинг инструктора пересчитывается на бэкенде и в дальнейшем отображается в карточках его тренировок (см. UC-3, UC-1).

### Альтернативные потоки
- **A1. Оценка без комментария.** На шаге 3 клиент указывает только звёзды, комментарий необязателен.

### Исключения / ошибки
- **E1. Тренировка ещё не завершилась.** Форма оценки недоступна до фактического завершения тренировки.
- **E2. Тренировка уже оценена.** Повторная попытка отправить оценку отклоняется (`already_rated`); клиент видит ранее поставленную оценку в режиме «только чтение».
- **E3. Сетевой сбой при отправке.** Мутация (отправка оценки) офлайн запрещена; при отсутствии сети — понятное сообщение, отправка недоступна; при неопределённом результате — единый Error/Retry.

### Постусловия
- Оценка сохранена и неизменяема; при успехе средний рейтинг инструктора обновлён на бэкенде.

## UC-5: Уведомление об отмене тренировки скалодромом

- **Актёры:** Клиент (получатель), Скалодром/бэкенд (инициатор отмены — вне скоупа клиентского приложения)
- **Предусловия:** У клиента есть активная бронь на слот; скалодром отменяет тренировку целиком (например, из-за профилактики зоны).
- **Триггер:** Слот отменяется на стороне существующей инфраструктуры (действие вне скоупа клиентского приложения).

### Основной поток
1. Скалодром отменяет слот (действие выполняется в существующей инфраструктуре, вне скоупа).
2. Бэкенд переводит все брони на этот слот в статус «Отменена скалодромом» с указанием причины.
3. Клиенту отправляется уведомление по обоим каналам: браузерный Web Push (если ранее было дано разрешение и подписка активна) и дублирующее сообщение в Telegram (если аккаунт привязан).
4. При открытии приложения (по клику на уведомление или самостоятельно) клиент видит статус «Отменена скалодромом» и причину в списке тренировок и в «Моих бронированиях»; кнопка повторной записи на слот неактивна.

### Альтернативные потоки
- **A1. Вкладка/браузер клиента закрыты в момент отмены.** Web Push доставляется операционной системой/браузером независимо от того, открыта ли вкладка приложения (через Service Worker), если подписка активна; при клике на уведомление открывается приложение. Если Web Push недоступен — статус брони в любом случае будет виден при следующем открытии приложения.
- **A2. Устройство/браузер не поддерживает Web Push (например, iOS Safari без установки на домашний экран).** Единственным доставленным каналом остаётся Telegram (если привязан); при его отсутствии клиент узнаёт об отмене только при следующем открытии приложения.

### Исключения / ошибки
- **E1. Web Push не доставлен (нет разрешения / отозванная или истёкшая подписка / браузер не поддерживает).** Уведомление по этому каналу не приходит, но статус брони обновлён на сервере и виден клиенту при открытии приложения; при наличии привязанного Telegram — дублирующее сообщение доставляется этим каналом.
- **E2. Telegram недоступен.** Дублирование не доставляется этим каналом; Web Push (если подписка активна) продолжает работать независимо (см. NFR-23).
- **E3. Попытка повторной записи на отменённый слот.** Отклоняется с кодом `slot_cancelled` (см. матрицу ошибок UC-1).

### Постусловия
- Бронь клиента находится в статусе «Отменена скалодромом» с причиной; повторная запись на слот заблокирована; клиент уведомлён как минимум одним доступным каналом (Web Push и/или Telegram), а при недоступности обоих — увидит статус при следующем открытии приложения.
