# Функциональные требования

> Этап 2. Что система должна делать (функции, поведение).
> Не путать с нефункциональными ([non-functional-requirements.md](non-functional-requirements.md)).

Приоритеты по MoSCoW: **Must** — обязательно к запуску; **Should** — желательно при наличии времени; **Could** — приятное дополнение; **Won't** — вне границ скоупа.

> **О нумерации требований.** ID требований (FR-NN) сохраняются стабильными для трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (лист ожидания, программа лояльности/отметка постоянных клиентов и онлайн-оплата рассматривались, но вынесены за пределы текущей поставки или в развитие). Нумерация намеренно не нормализуется, чтобы не ломать связи в таблице-источнике и смежных документах. Новые требования продолжают сквозную схему FR-NN без переиспользования занятых номеров.

## Границы скоупа (клиентское приложение + API)

Скоуп — **клиентское веб-приложение (React + Tailwind CSS) и API для него**. Функции инструктора, владелицы и
администратора обеспечиваются **существующей инфраструктурой** и здесь не описываются.

Следствия для скоупа:

- **Слоты (тренировки) приходят из существующего бэкенда через API.** Экранов создания и редактирования расписания в клиентском приложении нет.
- **Зоны/форматы и инструкторы** доступны клиенту как справочные (read-only) данные из API — клиент видит их в карточке слота, но не управляет ими.
- **Атомарность бронирования** (гарантия «0 двойных броней») обеспечивается на стороне бэкенда; клиент полагается на коды ответов API.

> **Граница с серверной интеграцией.** Детали серверной реализации брони (транзакции, SLA,
> маппинг полей) — вне скоупа клиентского приложения. Существующий бэкенд выступает black-box
> источником истины; атомарность операций и гарантия «0 двойных броней» обеспечиваются на его
> стороне. Клиент полагается на коды ответов и значения полей из ответа API.

---

> **Вне скоупа.** Функции инструктора, владелицы/администратора, CRUD расписания и админка
> обеспечиваются существующей инфраструктурой. Онлайн-оплата, лист ожидания и программа
> лояльности/отметка постоянных клиентов в клиентском приложении и его API не реализуются.
>
> **Лист ожидания — Won't (MVP).** Функция подписки на освобождение места при заполненном слоте
> осознанно не реализуется в текущей поставке. **Согласовано с заказчиком** (ответ № 3). На
> MVP-скоуп и трассировку требований это не влияет.
>
> **Отметка постоянных клиентов / программа лояльности — вне скоупа клиента.** Управляется
> только в существующей админке; в клиентском приложении не отображается и не реализуется.
> **Согласовано с заказчиком** (ответ № 14).
>
> **Онлайн-оплата — Phase 2.** На старте оплата (включая оплату проката) остаётся офлайн
> (наличные/перевод); приложение только фиксирует бронь и показывает тариф (см. FR-21).
> **Согласовано с заказчиком** (ответ № 10).
